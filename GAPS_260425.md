# Implementation Gaps — 2026-04-25

Status after ReWrite-1, ReWrite-2, and ReWrite-3.
Reference: `tree-sitter/lib/src/parser.h`, `crates/generate/src/render.rs`,
`crates/generate/src/build_tables/*.rs`.

---

## Current Remaining Gaps

### 1. Heavy C Parse-Table Construction Cost

`tree_sitter_c_json` now completes the routine compatibility path, including coarse
serialization, parser-table emission, parser.c emission, compatibility validation, and
compile-smoke. It is still expensive enough that it should stay in the heavy
compatibility step:

- serialize path is about 2-3 minutes locally
- peak RSS is roughly 3.4 GB
- it is not suitable for the fast default `zig build test` path

The next improvement should target parse-table memory pressure, especially item-set and
lookahead storage in coarse construction mode.

### 2. Parse-Table Pipeline Goldens Need EOF Refresh

Parser EOF/end-symbol work is implemented and covered by focused runtime tests, but the
older parse-table pipeline exact string goldens still reflect the pre-EOF dumps. The
compatibility-heavy step now filters to compatibility tests only, so these stale
pipeline goldens are isolated to `zig build test-pipeline`.

Refresh these goldens intentionally instead of treating the EOF actions as regressions.

### 3. Broad Real External-Scanner Promotion

RW-5 proves the basic external-scanner runtime surfaces: multi-token `valid_symbols`
routing, stateful scanner payload allocation/mutation/destruction, and a staged
grammar-json-to-parser-C external-scanner runtime link fixture.

Bash and Haskell remain blocked from `full_runtime_link` promotion by parse-table
construction cost and broader real-scanner fixture work, not by the basic generated
scanner-table wiring.

Generated nested bracket fixtures that require reduce-before-external-close behavior
still expose a parser-boundary gap. The staged RW-5 fixture therefore proves a flat
external-token sequence through the full local pipeline, while nested external-close
reduction remains future parser-table work.
